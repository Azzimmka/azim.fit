# KEEP AT IT — быстрый редактор и измеряемые тренировки

## Summary

Перестроить создание тренировки вокруг первого понятного действия — выбора упражнения — и добавить полноценные цели по повторениям, времени и дистанции. Пользователь не видит длинную форму: приложение открывает библиотеку, подставляет последние или рекомендуемые параметры, автоматически формирует название и показывает только поля, относящиеся к выбранному упражнению.

Режим активной тренировки расширяется timed-подходами и непрерывными кардио-сессиями. Для бега PWA считает GPS-дистанцию, активное время и темп в реальном времени, пока приложение видимо. Координаты и маршрут не сохраняются; при недоступном GPS всегда доступен ручной ввод.

## Goals

- Сократить число решений и полей при первом открытии редактора.
- Убрать повторный ручной ввод знакомых упражнений, подходов и отдыха.
- Поддержать упражнения на повторения, время и дистанцию без текстовых суррогатов.
- Различать подходы и непрерывные активности.
- Добавить production-ready GPS-сессию для бега в рамках реальных ограничений PWA.
- Сохранить совместимость с локальными и облачными данными существующих пользователей.

## Non-goals

- Подсчёт шагов.
- Сохранение карты, маршрута или истории координат.
- Гарантированное фоновое GPS-отслеживание при заблокированном экране или скрытой PWA.
- Интеграция с Apple HealthKit, Google Health Connect или носимыми устройствами.
- Временные и дистанционные личные рекорды в этом этапе.
- Публикация нативного приложения.

## UX flow

### 1. Старт создания

Нажатие «Создать тренировку» сразу открывает `ExercisePicker`, а не длинную форму. Первый экран содержит:

- поисковую строку;
- горизонтальный список недавних упражнений;
- категории встроенной библиотеки;
- действие «Создать своё упражнение».

После выбора первого упражнения открывается его компактная настройка. После подтверждения пользователь попадает в `WorkoutPlanBuilder`.

### 2. Выбор упражнения

`ExercisePicker` объединяет три источника:

1. встроенный локальный каталог популярных упражнений;
2. пользовательские упражнения;
3. недавние упражнения, вычисленные из истории пользователя.

Поиск использует существующую нормализацию названий: NFKC, нижний регистр, схлопывание пробелов и `ё → е`. Встроенные и пользовательские элементы отображаются вместе, но имеют стабильные разные идентификаторы.

### 3. Настройка упражнения

`ExerciseTargetEditor` показывает только поля выбранного типа. Значения подставляются в таком порядке:

1. параметры последней завершённой тренировки с этим упражнением;
2. параметры последнего сохранённого плана или шаблона;
3. рекомендуемые значения каталога.

Тип цели также задаётся каталогом, но пользователь может его изменить.

Примеры:

- отжимания: подходы + повторения, `3 × 10`, отдых `90 секунд`;
- планка: подходы + время, `3 × 180 секунд`, отдых `60 секунд`;
- бег: непрерывно + дистанция, `3000 метров`;
- бег: непрерывно + время, `1200 секунд`.

### 4. Компактный план

`WorkoutPlanBuilder` содержит:

- дату и время;
- карточки выбранных упражнений с кратким резюме;
- действие «Добавить упражнение»;
- свёрнутый раздел «Дополнительно»;
- закреплённую основную кнопку «Запланировать тренировку».

Название автоматически формируется из первых упражнений, например `Отжимания + планка`. Пользователь может изменить его вручную. В «Дополнительно» находятся ручное название, тип тренировки, интенсивность и повторяющаяся серия.

Пустую тренировку сохранить нельзя. Возврат из настройки упражнения не добавляет частично заполненный элемент.

### 5. Responsive behavior

- На телефоне picker и target editor открываются как полноэкранные нижние листы с `100dvh`, safe-area и закреплённым действием.
- На desktop те же компоненты открываются как центрированные диалоги.
- Touch-targets — не меньше `44 × 44 px`.
- На ширине до `360 px` числовые поля переходят в одну колонку.
- Открытие клавиатуры не скрывает основную кнопку и текущую ошибку.

## Domain model

### Exercise target

Упражнение получает явные поля вместо неоднозначного `plannedReps`:

```js
/**
 * @typedef {'sets'|'continuous'} ExerciseStructure
 * @typedef {'reps'|'duration'|'distance'} TargetKind
 *
 * @typedef {object} ExerciseTarget
 * @property {TargetKind} kind
 * @property {number} value
 * @property {'count'|'seconds'|'meters'} unit
 */
```

`Exercise` хранит:

- `structure`;
- `target`;
- `sets` только для `structure: 'sets'`;
- `restSeconds` только для подходов;
- `catalogExerciseId` или `customExerciseId` как необязательный источник;
- независимый snapshot названия и параметров.

Инварианты:

- `reps` использует целое `count` от 1 до 999;
- `duration` хранится целыми секундами;
- `distance` хранится целыми метрами;
- `continuous` не создаёт искусственные подходы и не использует отдых;
- `sets` допускает цели `reps` и `duration`;
- дистанция на этом этапе используется только для `continuous`.

### Results

Результат подхода хранит числовое `actualValue` в единице цели. Для continuous-активности используется один `ContinuousResult`:

```js
/**
 * @typedef {object} ContinuousResult
 * @property {'pending'|'completed'|'skipped'} status
 * @property {number|null} actualValue
 * @property {number|null} activeDurationSeconds
 * @property {number|null} averagePaceSecondsPerKm
 * @property {string|null} completedAt
 */
```

Координаты не являются частью `AppState`, reducer action, Firestore payload, логов или уведомлений.

### Progress and points

Единицей прогресса является один запланированный подход либо одна continuous-активность. Формула остаётся совместимой: `20` базовых баллов за завершённую тренировку и `+5` за каждую выполненную единицу. Timed-подход даёт `+5`; полностью завершённый бег без подходов также даёт `+5`. Skipped и pending единицы дополнительных баллов не дают.

Preview, progress bar, summary и awarded points используют одну чистую функцию подсчёта единиц. Временные и дистанционные PR не вычисляются; существующие весовые и повторные рекорды продолжают работать по мигрированным reps-результатам.

### Schema migration

Вводится `AppStateV3` с `schemaVersion: 3`, `customExercises` и `activeContinuousSession`. Новое состояние записывается в `keep-at-it-state-v3`; существующие `azim-fit-state-v2` и `azim-fit-state-v1` остаются нетронутыми.

Порядок чтения: scoped V3 → unscoped V3 → scoped V2 → unscoped V2 → V1. После успешного чтения старое состояние нормализуется и записывается в scoped V3 key. Повреждённый более новый key не мешает безопасно проверить предыдущий источник. Runtime-нормализатор принимает записи всех поддерживаемых версий.

- Числовой `plannedReps` мигрирует в `sets + reps`.
- Однозначные строки вида `30 сек`, `3 минуты`, `01:30` мигрируют в `sets + duration`.
- Неоднозначный текст сохраняется в необязательном `legacyTargetText` и не преобразуется молча в число; редактор попросит выбрать тип при следующем изменении.
- Старые `setResults[].reps` мигрируют в `actualValue` для `reps`.
- Исходный V2 localStorage key не удаляется.
- Firebase синхронизирует уже нормализованное состояние новой схемы.

Миграция идемпотентна и не меняет завершённые баллы или даты.

## Exercise library

Встроенный каталог — версионируемый JS-модуль без сетевой зависимости. Элемент содержит:

- стабильный id;
- русское название и поисковые aliases;
- категорию;
- рекомендуемые `structure`, `target`, `sets` и `restSeconds`;
- компактную иконку из существующей icon-системы.

Пользовательские упражнения хранятся в состоянии отдельно от тренировок. Созданное пользователем упражнение получает стартовые значения и появляется в поиске. Изменение или удаление библиотечного элемента не изменяет существующие тренировки.

Недавние упражнения и последние параметры вычисляются чистыми селекторами. Порядок определяется фактическим `completedAt`, затем временем создания плана. Завершённая тренировка имеет приоритет над незавершённым планом.

## Active session behavior

### Timed sets

- В приложении остаётся один глобальный `activeTimer`, которому добавляется `phase: 'work'|'rest'`. Одновременные рабочий таймер и отдых запрещены инвариантом reducer.
- Экран фокусируется на текущем подходе и показывает его длительность крупными табличными цифрами.
- «Начать подход» запускает обратный таймер.
- Доступны пауза и продолжение.
- На нуле звук и вибрация воспроизводятся best-effort, подход атомарно отмечается выполненным, затем запускается обычный отдых.
- Досрочное завершение требует подтверждения и сохраняет фактическую длительность.
- После reload таймер восстанавливается через абсолютный `endsAt`; истёкший таймер не пытается автоматически воспроизводить звук после холодного запуска.

### Continuous running

Сессия имеет состояния `idle → acquiring → active ↔ paused → summary → completed`.

- Разрешение геолокации запрашивается только после пользовательского действия «Начать бег».
- `watchPosition()` запускается с высокой точностью после выдачи разрешения.
- Первая принятая точка создаёт baseline и не добавляет дистанцию.
- Дистанция суммируется только между последовательными принятыми точками.
- Точки с плохой accuracy, невозможным скачком скорости, устаревшим timestamp или дубликатом отбрасываются чистым фильтром.
- Пороговые значения хранятся как тестируемые доменные константы, а не разбросаны по компонентам.
- При достижении цели включаются звук и вибрация, но сессия продолжается до ручного подтверждения.
- Пауза останавливает GPS watch и активное время.
- При `visibilitychange` в hidden сессия автоматически ставится на паузу; после возвращения показывается предупреждение о возможном пропущенном участке.
- Wake Lock запрашивается best-effort и не блокирует старт при отказе.

`activeContinuousSession` и запущенный `activeTimer` взаимоисключаются. При попытке начать второй активный процесс интерфейс предлагает вернуться к текущей сессии или явно завершить её; reducer не создаёт параллельное состояние.

Состояние восстановления хранит только accumulated meters, active duration, timestamps пауз и статус. Последняя координата остаётся только в памяти. После reload пользователь продолжает с новой baseline-точки либо вводит результат вручную.

Итог содержит:

- фактическую дистанцию;
- активную длительность;
- средний темп;
- дату завершения.

Маршрут и координаты уничтожаются при остановке watch, уходе со страницы, ошибке или завершении.

В смешанной тренировке continuous-упражнение занимает своё место в обычной последовательности. После его подтверждённого завершения активная сессия переходит к следующему pending-упражнению; отдельная тренировка с одним бегом сразу переходит к summary.

## Error handling

- Нет Geolocation API: сразу открыть ручной ввод.
- Permission denied: понятное сообщение с инструкцией браузера и ручным fallback.
- Permission timeout или position unavailable: повторить поиск сигнала или перейти к ручному вводу.
- Слабая точность: не увеличивать дистанцию; показывать статус «Слабый GPS-сигнал».
- Слишком долго нет валидной точки: предложить паузу или ручное завершение.
- Потеря видимости: автоматическая пауза, без фонового обещания.
- Ошибка сохранения Firebase: результат остаётся локально и повторно синхронизируется существующим sync-механизмом.
- Повреждённое target-состояние: runtime-нормализатор возвращает безопасный reps-default и фиксирует необходимость ручной проверки без падения UI.

Все ошибки отображаются через live-region. Валидация фокусирует первое проблемное поле.

## Architecture

Новые UI-границы:

- `ExercisePicker` — поиск, недавние и каталог;
- `ExerciseTargetEditor` — только параметры одного упражнения;
- `WorkoutPlanBuilder` — компактная композиция плана;
- `TimedSetSession` — таймер рабочего подхода;
- `ContinuousWorkoutSession` — orchestration GPS-сессии;
- `GpsStatus` — разрешение, сигнал и понятные статусы;
- `ContinuousResultEditor` — ручная коррекция итога.

Чистые доменные модули:

- target normalization и migration;
- форматирование seconds/meters/pace;
- exercise catalog search и recent ranking;
- default resolution;
- GPS point validation;
- Haversine distance и pace calculation;
- continuous session state transitions.

Компоненты не вычисляют distance и не принимают миграционные решения. GPS adapter инкапсулирует browser API и держит `lastAcceptedPoint` только в замыкании текущей страницы. Чистые функции валидируют точки и возвращают только delta meters и статус сигнала. Reducer получает агрегированные дельты и timestamps; сырые координаты не попадают в actions, React state, глобальный store или persistence.

## Accessibility and privacy

- Диалоги используют существующий focus trap и возвращают фокус на trigger.
- Переключатель типа цели реализуется как доступная radio-group или tablist с корректными ARIA-состояниями.
- Изменения GPS-статуса не перебивают каждое обновление дистанции; live-region сообщает только значимые события.
- Звук всегда дублируется визуальным состоянием и best-effort вибрацией.
- Пользователь видит отдельное объяснение, что маршрут не сохраняется.
- Geolocation требует HTTPS и явного разрешения пользователя.

## Testing

### Unit

- Миграция старых numeric и textual reps, malformed inputs и повторная нормализация.
- Валидация `reps`, `seconds`, `meters`, sets/continuous invariants.
- Автоматическое название и глубокое копирование библиотечных элементов.
- Поиск, aliases, recent ranking и приоритет последней завершённой тренировки.
- Seconds/meters/pace formatting и русские формы.
- Haversine distance, accuracy filter, duplicate points, impossible speed, pause gaps.
- Continuous state machine, idempotent finish, reload baseline и manual correction.
- Timed-set expiry, early finish, pause/resume и timer restore.

### Component

- Picker search, recent items, categories and custom exercise.
- Progressive disclosure и focus management.
- Target-specific fields и validation.
- Compact plan, automatic title и advanced section.
- Timed-set UI and rest transition.
- GPS permission states, weak signal, hidden-page pause и manual fallback.
- Privacy assertion: raw coordinates never reach store callbacks.

### E2E

- Создать отжимания из недавних значений.
- Создать планку `3 × 3 минуты`, выполнить timed-подходы и отдых.
- Создать бег по времени и по дистанции.
- Смоделировать GPS points, пройти цель, завершить и сохранить только итог.
- Permission denied, weak signal, visibility pause, reload and manual correction.
- Offline installed-PWA flow и последующая Firebase synchronization.
- `390 × 844`, ширина до `360 px`, `1440 × 900`, keyboard and overflow checks.

Финальный gate: `npm run check`, `npm run test:e2e` и ручной smoke-test установленной PWA на телефоне.

## Implementation sequence

Работа выполняется тремя последовательными production-gates внутри одного плана:

1. `AppStateV3`, миграция, библиотека и новый progressive editor.
2. Reps/duration targets, timed-set session и совместимые результаты.
3. Continuous time/distance targets, GPS adapter, running session и privacy verification.

Каждый gate завершается lint, unit/component tests и production build. E2E добавляются по мере появления пользовательского потока; полный PWA smoke-test выполняется после третьего gate.

## Platform constraints

`watchPosition()` требует secure context и явного разрешения пользователя. Спецификация геолокации не гарантирует обновления, когда документ не видим, поэтому фоновой GPS-трекинг не заявляется как поддерживаемая функция. [MDN `watchPosition()`](https://developer.mozilla.org/en-US/docs/Web/API/Geolocation/watchPosition), [W3C Geolocation](https://www.w3.org/TR/geolocation/).

Шаги не входят в продукт: Web API предоставляет низкоуровневое движение устройства, но не надёжный системный pedometer. Точный доступ к step count относится к нативным health frameworks. [W3C Generic Sensor API](https://www.w3.org/TR/generic-sensor/), [Apple HealthKit stepCount](https://developer.apple.com/documentation/healthkit/hkquantitytypeidentifier/stepcount).

## Acceptance criteria

- Новый пользователь может создать первую тренировку, не видя длинной формы.
- Повторное добавление знакомого упражнения требует выбора и одного подтверждения.
- Отжимания, планка и бег имеют семантически корректные цели и результаты.
- Ненужные поля никогда не показываются для выбранного типа.
- Timed-подход автоматически завершается на нуле и переходит к отдыху.
- GPS-сессия показывает дистанцию, active time и pace в реальном времени при видимой PWA.
- При недоступном GPS тренировка завершается вручную без потери плана.
- Ни одна координата не сохраняется локально или в Firestore.
- Существующие тренировки читаются без потери данных.
- Все production gates проходят.
